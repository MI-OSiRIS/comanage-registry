Quick Reference for Ceph S3/Rgw Access Key

Userid

Identifies the owner of data buckets. Is not given to S3 clients for authentication but may be used to set bucket ACL allowing other userid access to a bucket. You can create additional user id if you would like to isolate data buckets for different usage. Buckets created by one userid are not by default accessible to your other user id.

You can ask that any one of your S3 userid be mapped to a CILogon certificate DN for used with Globus, but only one can be mapped to any given identity. It can be changed by request. If you have multiple institutional identities with different certificate DN we can map each one to a different user id.

access_key and secret_key

Used for authentication by S3 client. You can create as many of these as you like and delete them if they become out of date. You cannot delete your last access/secret pair but instead use the 'regenerate' button to make a new pair if you want to reset it.

All access_key associated with a userid by default will see the same view of S3 data buckets. Buckets created using any given set of access_key / secret pair are associated with a single user id. Using any of the access_key for a given user id sees the same view of S3 buckets and has same access rights - permissions are associated with the user id, not with the access/secret pair.

Downloading credentials
The download button will download the credential in a format suitable for saving directly as ~/.aws/config. The python boto client will then read this file for authentication.

For more assistance and URLs to the S3 endpoints please look at our documentation page.

You may also want to look into Globus for access to your S3 storage

Quick Reference for Ceph Client Key

These are credentials for direct connection to the Ceph cluster for using Rados or RBD. Every COU has 2 pools created by default: cou.YourCouName.rados and cou.YourCouName.rbd. Everyone in the COU can access these pools using credentials obtained from the Ceph credentials page. Most users will not directly access the cluster in this way but the option is available.

Userid

Ceph client keys are identified as 'client.id'. The userid printed here is the 'id' portion.

Credentials

The shared secret key.

Downloading Credentials

The download button will download the credential in ceph client keyring format. Typically ceph clients lookup these keyrings in a path like /etc/ceph/ceph.client.osiris.yourid.keyring.

For direct cluster connections you'll need to contact us to open up appropriate network ports. We have some additional information documented here.

Using CephFS or shell/scp access

These credentials are not required.
